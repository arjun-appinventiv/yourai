## Product overview

**YourAI** — private, single-tenant AI platform for US law firms. Attorneys and paralegals can chat with legal documents, run structured workflows (multi-step AI pipelines), search a curated knowledge base, and get outputs like summaries, risk memos, clause analyses, timelines, and research briefs.

Two user surfaces, one codebase:
- **Tenant app** (`/chat`, `/chat/workspaces/:id`, etc.) — what end users see
- **Super Admin portal** (`/super-admin/*`) — YourAI staff administer tenants, users, global KB, bot persona
- **Org Admin portal** (`/app/*`) — tenant admins manage their own org

Deployed at `yourai-black.vercel.app` (production main deploys from the `yourai` remote's `main` branch; `origin` remote is `scope-creator-ai` — stale, do not push there).

## Tech stack

- **Frontend**: Vite + React (18) + TypeScript (mixed with legacy .jsx) + shadcn-ui (Radix primitives) + Tailwind CSS
- **Auth + role**: context-based (`src/context/AuthContext.tsx`, `src/context/RoleContext.tsx`); localStorage-backed session
- **Routing**: React Router v6, `BrowserRouter`
- **Backend**: Vercel Edge Functions (`api/*.ts`) proxy to OpenAI; an in-repo FastAPI-ish `backend/` folder (Prisma + SQLite `dev.db`) exists but production chat does NOT use it — it hits the Edge function directly
- **LLM**: OpenAI gpt-4o-mini via `/api/chat` (server-side proxy, never exposes key to client)
- **State**: React state + localStorage mock-API pattern (every mock function is shaped like the REST endpoint it will eventually become)
- **Styling**: design tokens in `src/index.css` CSS variables (`--navy`, `--gold`, `--ice-warm`, `--border`, etc.); DM Serif Display for titles, DM Sans for body, IBM Plex Mono for labels

## High-level architecture

- **Chat flow**: user types → `ChatView.jsx` classifies intent via keyword matcher → POSTs to `/api/chat` with `{message, history, intent, ...}` → Edge function injects per-intent JSON schema when intent is a "card intent" and calls OpenAI with `response_format: json_object` → client streams the response, parses JSON for card intents (fallback to markdown on parse failure), renders via `IntentCard` dispatcher.
- **Workflows**: multi-step AI pipelines run by a module-level singleton (`src/lib/workflowRunner.ts`) that survives component unmount via a subscribe/unsubscribe pattern. Runs persist in localStorage under `yourai_workflow_runs_v1`.
- **Knowledge Packs + Document Vault**: independent session-level state on `ChatView`. Neither is auto-attached by intent. Content gets injected into the system prompt at send time (client fallback path only; Edge path receives IDs but not content).
- **Role model**: `SUPER_ADMIN` (YourAI staff), `ORG_ADMIN` (firm admin), `INTERNAL_USER` (firm attorney), `EXTERNAL_USER` (client). External users have no workflow access, no `/chat` home — they land in workspace chat only.

## Conventions

- **File naming**: `PascalCase.tsx` for components, `camelCase.ts` for libs, `kebab-case.md` for docs.
- **Legacy .jsx**: don't migrate to .tsx unless changing substantial logic; it churns diffs. Keep mixed.
- **Comments**: only explain the *why* — hidden constraints, non-obvious invariants. Never narrate what code obviously does.
- **Feature flags / backwards-compat**: avoid. When removing something, delete it; don't leave `// removed because X` tombstones.
- **Storage keys**: all localStorage keys prefixed `yourai_*` and versioned with `_v1`/`_v2` suffixes to dodge collisions with prior iterations.
- **Routes**: tenant chat `/chat/*`, workspace chat `/chat/workspaces/:id`, org admin `/app/*`, super admin `/super-admin/*`, auth `/chat/login`, `/chat/signup`, etc.
- **Error handling at boundaries only**: trust internal code. Validate only at user input / external API surfaces.
- **Icons**: `lucide-react`. When you use `<SomeIcon>` in JSX, *make sure it's in the import* — repeated production blanks have been caused by missing lucide imports.
- **CSS-in-JS**: inline `style={{}}` is used heavily; Tailwind is used selectively. Both are fine.
- **Tests**: none automated — QA via `YourAI-Test-Suite.md`.
- **AI calls for new features go through `/api/chat` (Edge), not `callLLM`**. `callLLM` (`src/lib/llm-client.ts`) is the client-side fallback path that requires a browser-side `VITE_OPENAI_API_KEY` which is NOT set in production. As of 2026-04-25 **no production code path uses `callLLM`** — `workflowExecutor`, main `ChatView`, and `WorkspaceChatView` all hit `/api/chat` directly. Anything new must `fetch('/api/chat', { method: 'POST', body: JSON.stringify({ messages, model, temperature, max_tokens }) })` (or the simpler `{ message, history, system }` shape). Symptoms of regressing back to `callLLM`: bot says **"Something went wrong reaching the AI"** (workspace chat error path) or **"No LLM backend available. Please configure the API key…"** (the latter copy is now removed but watch for it returning).
- **Card empty-state pattern** — every intent-card component must detect "schema-shaped envelope with no real data" and render a friendly empty-state in its normal shell instead of grids of `—` dashes. The Edge forces `response_format: json_object` for card intents, so the LLM will return valid empty JSON when the user supplied no document. Detection rule: **all schema-required string fields blank AND no `documentName` AND empty arrays** → render upload prompt + sibling-intent hint. See `.claude-context/card-empty-state-pattern.md` for the full template and the per-card detection-and-message specifics.
- **Edge `MISSING DOCUMENT HANDLING` branch** in `api/chat.ts` system prompt — when a document-analysis intent comes in without a document, the prompt explicitly instructs the model to ask for an upload (echoing the user's task back, ≤50 words) instead of falling through to the generic off-topic refusal. Companion to the card empty-state pattern: the LLM picks prose vs. JSON unpredictably, so both layers must exist.
- **"Not covered by supplied documents." is a literal, verbatim anti-hallucination sentence** used across every workflow operation's system prompt. When the uploaded docs don't contain what a step needs, the step's output must begin with that exact sentence + a one-sentence reason, then produce whatever partial analysis is possible. QA tests check for the literal string — don't rephrase.
- **Workflow operation prompts live in `src/lib/workflowPrompts.ts`**. When adding a new operation or editing an existing one, update that file (not scattered prompt strings elsewhere). Execution is always via `src/lib/workflowExecutor.ts` → `/api/chat`.
- **Card-in-card rendering uses a `variant="embedded"` prop** to suppress the nested card's outer chrome (border, radius, shadow, accent stripe, header). See `WorkflowProgressCard` as the canonical example.
- **One primary CTA per screen** — wizard pages especially should not have both a top-bar Continue/Save AND a bottom-row equivalent.
- **Additive uploads (replaces DEC-095 Option C)** — the prior "one upload per chat" rule is gone. New docs uploaded mid-thread APPEND to `sessionDocContext`, get labelled `Document N (added HH:MM): filename` in the system prompt, and trigger an inline thread note offering a one-click "Start a new chat →" escape hatch (window event `yourai:start-new-chat` → ChatView's top-level listener calls `handleNewThread`). When touching the upload pipeline, do NOT reintroduce a block — the friction was named by the attorney user (Wendy 2026-04-27 interview).
- **Doc content MUST be inlined into the `/api/chat` message body** — the Edge function only sees `body.message` + `body.history` + `intent`. It does NOT have access to `contextLayers` or any side-channel for attachments. Every send-pipeline addition that needs the model to see attached file text must stitch it into `messageForEdge` under a `[Documents attached to this conversation]\n…\n\n[User question]\n` header. Same pattern in `ChatView` and `WorkspaceChatView`. Symptom of regressing: bot replies "I'd be glad to review that document. Upload using the + button…" even though a doc is visibly attached.
- **`closeAllPanels()` helper for full-page panel mutual exclusion** — `ChatView.jsx` defines a single helper that zeroes every `show*Panel` flag (Team / Workspaces / Workflows / Prompt / Clients / KnowledgePacks / DocumentVault / editingWorkflow). Every Sidebar nav-item open-handler must call it before setting its target flag — otherwise two full-page panels render side-by-side as flex peers (real bug seen 2026-04-27). The "Open YourVault →" / "Manage knowledge packs →" footer links inside the chat-input pickers also call it.
- **Sidebar `active` state is dynamic, derived from `sidebarActiveKey`** — never hard-code `active: true` on a nav item. The active key is computed from the show*Panel flags inside `ChatView`'s main component; full-page panels take precedence over the underlying chat in the precedence chain. Sidebar item objects then `.map((it) => ({ ...it, active: it.id === activeKey }))`.
- **General Chat is the post-login landing surface for non-external users** — there is no `/chat/home` route; the prior tile-launcher home page was retired 2026-04-30. Login / signup-invited / onboarding-complete all navigate straight to `/chat`. Externals still land on `/chat/workspaces`. A `<Navigate to="/chat" replace>` route catches stale `/chat/home` bookmarks. Do NOT add a Home sidebar entry, a `/chat/home` element route, or an `initialView="home"` branch back — the chat empty state's intent-pill row + drop zone + `+` attach picker are the canonical entry points now.
- **`+` searchable picker is the only YourVault attach affordance on the empty-state input** — single button, single popover, single mental model. Always-on autofocus search input + filtered list of every vault doc (substring match on `name + description + fileName`) + Detach row (only when one is pinned) + "Open YourVault →" footer routing to the full Vault page. State: `isVaultAttachOpen` + `vaultAttachQuery` (cleared on close). Picking a doc sets `activeVaultDocument`; sendMessage's existing `vaultSelectionContext` branch inlines the doc's full text. Do NOT add: per-item submenus, scope dropdowns, AI-auto-retrieve toggles, or a side-by-side second affordance — three rejected designs are on the do-not-reintroduce list. If a "search across all my docs" need re-emerges, ship it as a Vault-page action or a silent-citation pattern, not as a chat-input toggle.
- **All pickers that list user-owned items use the search-first pattern** — sticky search input at the top (autofocus, always on for the vault picker; >5-item threshold for the KP picker), filtered list below (substring match), footer link to the full management page. Applied to: vault picker, KP picker. Do NOT reintroduce flat-list popovers without a search input on libraries that can grow past ~10 items — the listing problem comes back. State: `vaultAttachQuery` / `kpQuery`, both clear on close.
- **Workspaces is sidebar navigation, not a chat-input control** — was tried briefly inside the (also-retired) Search Within dropdown. Pulled because cross-workspace search from `/chat` is a confidentiality footgun for law firms (matter privilege between cases). Switching matters happens via the sidebar Workspaces entry → workspace picker → workspace chat surface; each workspace chat already RAGs within its own corpus.
- **Intent dropdowns render verb buckets, not flat lists** — `src/lib/intents.ts` exports `INTENT_BUCKETS` (DEFAULT / ASK & RESEARCH / ANALYZE / DRAFT) + `groupIntentsByBucket(ids[])` which filters empty buckets and preserves canonical order. Both the populated-chat collapsed-pill dropdown and the empty-state "More operations" overflow render from this. Bucket label = uppercase 11 px IBM Plex Mono / 700 weight / `--text-primary` with 0.14em tracking; non-first buckets get a navy 2% wash + 1 px top-border. When adding a new intent: register it in `INTENTS`, then add its id to the appropriate bucket in `INTENT_BUCKETS` (don't leave it bucket-less — it'll silently disappear from both dropdowns).
- **Vault search paths — there's now only one.** The YourVault page toolbar search checks `name + description + fileName` (NOT content) — the substring filter in `filteredDocs` is one line away from including `d.content` but client-side substring across N×100K-char strings gets laggy at scale; defer until backend lands. The chat-input vault picker uses the same matching against the same fields. The earlier `vaultScopeContext` token-overlap retrieval in `sendMessage` was retired with the Search-my-docs toggle — if "search across all my docs" comes back, design it as a separate flow (silent retrieval + inline citations, or a Vault-page action) and don't bring back the toggle.
- **Chat-attached files auto-save into YourVault — including extracted content.** `handleAttachFiles(kind='doc')` creates a vault entry on attach (metadata only, deduped by fileName via `addedFromChat: true`); the extraction `.then()` then backfills the entry's `content` field via the `vaultIdByFileName` map. Without this backfill, the doc would live in the vault as metadata-only and the picker would still find it by name but the chat couldn't actually read it. If a vault entry already has content (richer prior copy), skip the overwrite.
- **Drop zone is persistent across the conversation** — both prior gates (`showEmptyState` AND `pendingAttachments.length === 0`) are gone. The tile renders below the input throughout the chat, with reduced padding in populated state. Label flips to `Add another file (N attached)` when files exist; the chip row above the input handles per-file removal via X button. The strip's `· saved to YourVault` helper text is the only visible reminder that uploads persist.
- **Chat-input textarea uses the visually-hidden scrollbar pattern** — `.no-focus-ring` class includes `::-webkit-scrollbar { width: 0 }` + `scrollbar-width: none`. Without these, macOS users with "Always show scrollbars" preference see a persistent dark line between the textarea and the KP dropdown even at empty content. Scroll still works via mouse-wheel + arrow keys when content overflows the maxHeight cap.
- **Verb on a button beats noun in a dropdown for chat-input controls** — design lesson from the 2026-04-30 attachment iteration. "Scope" / "Source" / "Search Within" are all engineer-speak nouns; users think in verbs ("attach a doc", "find the MSA"). When a control's job is to let the user act, prefer a `+` icon, an action verb, or a deterministic picker. Reserve abstract noun-mode dropdowns for user-pinned settings, not per-message decisions.
- **Ask-anything NL parser sets *transient* filters** (`resultLimit`, `askExplanation`, `sortBy`) that **clear on user typing or All-documents click** — explicit chip filters (date / uploader / type) stay set because those are deliberate user choices. Don't conflate the two when adding new filters: chip = sticky, parser-set = transient.
- **Card-intent results render in a Claude-style right-rail artifact panel as MARKDOWN, not as IntentCard chrome** (2026-05-04). Card responses (Risk Memo, Summary, Comparison, Case Brief, Research, Clause Analysis, Timeline) show a compact preview chip in chat + a 540 px `IntentArtifactPanel` (sibling of chat-main; chat shrinks). Panel body uses `ReactMarkdown` rendering of `cardDataToMarkdown(intent, data)` from `src/lib/cardToMarkdown.ts` — h1 + meta line + h2 sections + bullets + plain blockquotes. NO card chrome (no DOCUMENTS ANALYSED eyebrow, no document tile, no severity pills, no gold rails). State: `activeArtifactMsgId` on `ChatView`. Auto-opens on new card response (and on `/demo-*` slash commands). Click chip → reopens. **`find_document` keeps inline FileResultsCard rendering** — search results read better in the conversation flow; do not move it into the panel. When adding a new card intent, write the per-intent serializer in `cardToMarkdown.ts` first; the panel picks it up automatically. Full pattern doc: `.claude-context/artifact-panel-pattern.md`.
- **`pickTitle(d, fallbackLabel)` rejects generic LLM-emitted titles** (`Risk Assessment of Uploaded Documents`, `Untitled case`, `Legal Inquiry`, `This contract`, etc.) and falls back to documentName / sourceName / intent label. All seven serializers in `cardToMarkdown.ts` route through it. When adding a new serializer, use `pickTitle(d, 'Whatever fallback')` for the H1 — never trust raw `matterName` because the LLM hallucinates generics for trivial prompts. Pattern: `GENERIC_TITLE` regex covers the common ones; add to it if you spot a new generic.
- **Chit-chat on card intents is routed through the LLM via a `general_chat` override**, not via static replies. When user picks a card intent + types chit-chat ("hi" / "how are you" / "what can you do") + has no doc attached, `sendMessage` flips `edgeIntent` to `'general_chat'` (so the Edge does NOT force JSON via `response_format`) and prepends a context preamble to `messageForEdge` naming the active card intent + telling the LLM "no document attached yet — respond warmly + remind them what to upload, prose only, ≤80 words". User's chat bubble shows the original message; only the Edge sees the augmented version. Bot-message metadata flips to `intent: 'general_chat'` when override fires so downstream filters / card dispatchers don't try to render this as structured. Detection: `CHIT_CHAT_RE` covers ~40 patterns + `length ≤ 60 chars AND no analysis verb` fallback. Don't reintroduce static-string replies — they read robotic by the third turn.
- **Doc-source confirmation when card intent + already-attached docs** — bot pauses and asks before running the analysis. Rendered as **plain prose with two inline `<a>` text links** ("Yes, use it" / "I'll upload a new one"), NOT a styled card with pill buttons. PM read: should look like part of the chat, not a UI component. Click "Yes" → re-fires `sendMessage(text, { skipDocConfirmation: true })`. Click "Upload new" → swaps message to "OK — drop the new document via the + button…". **The gate uses `willBeCardIntent`** (computed via `detectIntent` look-ahead) instead of the live `activeIntent`, so the confirmation also fires when activeIntent is still general_chat but the message will auto-switch ("Do clause analysis of attached doc"). Without that look-ahead, the auto-switch fires AFTER the confirmation gate and the bot silently picks docs. When adding any flow that runs an analysis on attached docs, plumb `skipDocConfirmation` through if the user already confirmed.
- **`extractionPromisesRef` — await text extraction before `sendMessage` builds the Edge body.** `extractFileText` is async; without awaiting, the first send ships a `[File: …] Text extraction is still in progress…` placeholder and the LLM reads it as "no doc attached" → upload prompt fires. `handleAttachFiles` stores each `extractFileText` promise in `useRef<Map<id, Promise<text>>>()`. `sendMessage` awaits unresolved promises (12 s max) for the current `pendingAttachments` then **re-resolves content from the promise return values** (state may not have applied within the same async tick — read from the promise, not from `pendingAttachments[i].content`). When adding new code paths that read attachment content, account for this ref.

## Gotchas (non-obvious things)

1. **Two git remotes**: `origin` → `scope-creator-ai` (stale), `yourai` → live production repo. Always push to `yourai`. Production `yourai/main` is what Vercel deploys.
2. **Production branch flow**: working branch is `claude/great-banach`. Ship via merge-to-main: `git fetch yourai main && git checkout -B tmp yourai/main && git merge --no-ff claude/great-banach && git push yourai tmp:main`. Branch-name fine; the merge-to-main ritual is what matters.
3. **No CI for undefined-symbol errors**: missing `lucide-react` imports blow up the whole app with `ReferenceError` at render. Build passes (TS doesn't check JSX symbol scope). Always manually verify new icon imports. Past incidents: `listRuns` and `ArrowRight` both blanked production.
4. **Edge function ≠ client fallback**: `/api/chat` (Edge) and `src/lib/llm-client.ts` (client) have *separate* system prompts. Changes to one do not propagate to the other. The client fallback only runs when the Edge is unreachable.
5. **Bot Persona config only affects client fallback**: SA-edited system prompts in `yourai_bot_persona` localStorage never reach the Edge function. Production responses use the Edge's hardcoded prompt + optional card schema.
6. **3 sources of truth for intents** (see `docs/extracted/intent-features.md` §8): `intents.ts` + `intentDetector.ts` + `GlobalKnowledgeBase.jsx` DEFAULT_INTENTS. They drift. When adding an intent, update all three.
7. **Intent ↔ Knowledge Pack has zero wiring**: changing intent never changes the active KP or vault doc. They're orthogonal session-level states.
8. **Headless-Chrome smoke test pattern**: when you think production might be broken after a deploy, run `curl -s <url>` to get the bundle hash, then use `/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --headless --dump-dom <url>` with `--enable-logging=stderr` to catch `ReferenceError`s. This caught both blank-production incidents.
9. **ChatView.jsx is 4000+ lines**: grep ruthlessly, don't Read whole-file. Key sections: state (~2780), auto-switch (~3005), card dispatch (~3220), suggestion banners (~3900), intent dropdown (~3970).
10. **CSS grid `auto-fill` creates ghost columns on sparse data**: a grid with `repeat(auto-fill, minmax(300px, 1fr))` at a 1440px viewport with only 2 cards leaves a ~40% blank band on the right. For sections that often render few items, use `auto-fit` (collapses empty tracks) and/or cap `maxWidth` on the grid container. Aashna flagged this on the Workflows picker; fix is documented in the session transcript.
11. **Empty states: anchor via `padding-top`, not flexbox centering**: a container with `flex: 1; justify-content: center` or `align-items: flex-end` produces huge dead zones. Use `padding-top: 14vh` (approximately 15% from the top) inside a fixed max-width centred column instead. This pattern now ships on the chat empty state.
12. **DM Serif is decorative-only**: reserve it for hero titles and section headers. Never use it for live status strings or dynamic counts ("2 running · 3 recent") — those should be DM Sans. Run Panel header once broke this rule and read as decorative when it needed to read as data.
13. **Workflow executor uses `/api/chat` with a full `messages[]` payload**: each step = operation system prompt + assembled user message (step metadata + user instruction + prior step outputs capped at ~3500 chars each + uploaded docs capped at ~8000 chars each + optional reference doc + workspace context). Streams via `ReadableStream`, 90s timeout with `AbortController`. See `src/lib/workflowExecutor.ts`.
14. **Prior-step chaining is the whole point of a workflow**: step N receives a structured summary of every completed step 1..N−1. The Generate Report step is a synthesis of prior outputs, not a re-analysis. Don't send only the raw documents to step N without the prior outputs.
15. **CSS grid for picker cards uses `auto-fill, minmax(340px, 1fr)` with no `maxWidth` cap on the section** — aashna's chat-mode mockup expects 3-across at desktop reflowing naturally to 2 / 1 on narrower screens. The earlier `auto-fit, minmax(280px, 340px)` with a `maxWidth: 960` cap fought the design at every viewport. Don't reintroduce the cap unless aashna explicitly asks.
16. **Card-rendering intents force JSON via `response_format: json_object`** even when the user attached no document. The LLM will dutifully return a schema-shaped envelope with empty strings and empty arrays. Without the empty-state detection (Conventions §card empty-state pattern) the card renders as a grid of `—` dashes — the chat reads as broken even though the LLM did exactly what was asked.
17. **`credentials: 'include'` on same-origin fetches is vestigial and can trigger Edge / cache weirdness**. Drop it on any `fetch('/api/chat', …)` call. The Edge doesn't read cookies.
18. **Mac screenshot filenames sometimes contain U+202F (narrow no-break space)** instead of ASCII space between minute digits and "PM". `ls` shows them but `cp` / `Read` fail with "No such file". Workaround: copy via shell glob (`Screenshot 2026-04-24 at 1.18*.pdf`) or rename via `tr -d $'\xe2\x80\xaf'`. Hit this twice when reading aashna's mockup PDFs.
19. **Streaming reader needs a trailing `decoder.decode()` flush** after the `while`-loop ends. Without it, multibyte UTF-8 characters split across chunk boundaries lose their final byte. `ChatView`, `WorkspaceChatView`, and `workflowExecutor` all do this — match the pattern when adding new streaming consumers.
20. **React Router routes that share a component with different props don't always remount** — when navigating between `/chat` and `/chat/workspaces` (both rendered by `<ChatView>` with different `initialView` props), the `<ChatView>` instance can be reused. Any state initialised from `useState(initialView === 'workspaces')` will NOT re-init on the new prop. Fix: handlers that route between same-component routes must `setShowXxxPanel(true)` explicitly before navigating. Bit the (now-retired) home tile's `onOpenWorkspaces` 2026-04-27 — *clicking the tile sent the URL to `/chat/workspaces` but the panel never opened* because the `closeAllPanels`-set false stuck. Same risk applies any time a future route is added that shares ChatView.
21. **Vercel deployment protection returns 401 with `{"error":"Not authenticated"}` JSON** for preview-branch URLs (`*-claude-great-banach.vercel.app` etc). Production URL `https://yourai-black.vercel.app` is unprotected. If a tester reports "AI service returned 401. Not authenticated" — first thing to check is whether they're on a preview URL.
22. **`fpdf2.multi_cell()` leaves cursor x at the right margin** — the next `multi_cell(0, h, line)` interprets `w=0` as "remaining width from current x", which is 0, and throws "Not enough horizontal space to render a single character". Fix: `pdf.set_x(pdf.l_margin); pdf.multi_cell(pdf.epw, h, line)` before each call. This bit me writing `/tmp/gen-sample-pdfs.py` — relevant if anyone re-runs that PDF generator.
23. **Default vault seed key is `yourai_document_vault_v2`** (bumped 2026-04-27) — older entries didn't carry `content` or `sampleUrl`. Bumping forces re-seed so users get the new sample PDFs and the bot can actually read them. If you add new fields to seed entries that should reach existing testers, bump again (`_v3`).
24. **Sample seed doc content lives in `src/data/sampleVaultContent.ts`**; the matching real PDFs in `public/sample-docs/` are generated by `/tmp/gen-sample-pdfs.py` (uses `fpdf2`, install via `python3 -m pip install --user fpdf2`). When the sample text changes, re-run the script — it parses the .ts file, regenerates the PDFs, and writes to public/. Vite copies public/ into dist/ on build with no extra wiring.
25. **(retired 2026-04-30)** ~~Home tile launcher render path~~ — the `/chat/home` tile launcher was deleted; `HomeTileLauncher` and the `initialView === 'home'` branch are gone from `ChatView`. Login default for non-externals is now `/chat`. A `<Navigate to="/chat" replace>` route in `App.jsx` catches stale `/chat/home` bookmarks.
26. **Programmatic `setState` does NOT fire DOM events** — `setInput(promptText)` updates a controlled-textarea value but does NOT fire `onInput`. Any auto-resize / auto-focus / auto-anything that lives on `onInput` will silently miss programmatic updates. Pair every controlled-textarea with a `useEffect` that watches the value and re-runs the side-effect (e.g. `inputRef.current.style.height = 'auto'; inputRef.current.style.height = Math.min(inputRef.current.scrollHeight, 140) + 'px';`). Bit us 2026-04-29 when action-pill pre-fills clipped at one row because `onInput` never fired.
27. **Auto-switch logic must respect manual user picks** — when an auto-detector (e.g. keyword-based intent classification) can override user state, gate it on a `hasManualXxxPick` flag. Set the flag wherever the user explicitly picks (every place that calls `setActiveIntent` from a click — pill rows, dropdowns, suggestion-banner accepts). Clear it on new-thread / new-session. Without this, a deliberate pick gets silently overridden by the auto-detector on the next send. Bit us 2026-04-29 — clicking "Review a contract" set the intent visually but `sendMessage`'s auto-switch path saw `activeIntent === 'general_chat'` (race or re-pick) and overwrote the user's choice mid-send.
28. **Vercel webhook can silently miss a deploy** — push lands on `yourai/main` but the prod bundle hash doesn't change. Diagnose by `curl -s https://yourai-black.vercel.app/chat | grep -oE 'index-[A-Za-z0-9_-]+\.js'` against the expected new hash. Recovery: push an empty commit (`git commit --allow-empty -m "chore: retrigger Vercel"` then merge to main and push) — that nudges the webhook. Hit this 2026-04-29 between the capability-grid push and the eventual flip.
29. **Vercel bot-challenges aggressive curl** — repeated automated requests from the same IP get a 403 with `x-vercel-mitigated: challenge`. Affects automated checks only; user browsers are unaffected. If verifying prod after deploys repeatedly, space out the polling, or hit through headless Chrome with a `Mozilla/5.0` User-Agent.
30. **Sub-agent sandbox blocks `git`, `curl`, headless-Chrome on the great-banach worktree path** — local-agent sub-agents in this harness can do file edits + `npm run build` but cannot commit, push, or run smoke tests against the dev server. Plan accordingly: parent agent commits + deploys + smoke-tests after the sub-agent reports back. Bit multiple sub-agents 2026-04-28 → 2026-04-29.
31. **When retiring a state variable, grep for ALL read sites — not just the assignment.** The `vaultScopeContext` token-overlap retrieval branch was retired with the 2026-04-30 Search-my-docs cleanup, but its read site at the "Final assembly" step in `sendMessage` (`const effectiveDocContext = mergedDocContent || vaultSelectionContext || vaultScopeContext;`) survived the cleanup. Result: every send threw `ReferenceError: vaultScopeContext is not defined` inside the click handler, the user msg appeared in the thread but the bot reply never mounted, and the symptom was "the chat doesn't work" across every intent including General Chat. CDP probe driving Chrome was what finally surfaced it. When deleting a state field, grep for `<name>` not just `set<Name>` / `<name> =`. Hit 2026-05-04. The fix landed in commit `9d0c8cc`.
32. **CDP probes for testing live prod** — when behavior is hard to reproduce visually, drive Chrome via CDP. Pattern: spawn `--headless=new --remote-debugging-port=9XXX --user-data-dir=/tmp/...`, fetch `/json/list`, find the page target, open a WebSocket, send `Runtime.evaluate` with `returnByValue: true`. Listen for `Runtime.exceptionThrown` events for uncaught errors. The dev `ws` package isn't always installed (it's a transitive dep that gets pruned by `npm install` cycles); install it with `npm install --no-save ws` from inside the worktree if your probe fails with `Cannot find package 'ws'`. Live probes against prod were how we caught the `vaultScopeContext` ReferenceError, the artifact-panel chip not opening on click, and the doc-source confirmation not firing on auto-switched intents. **Never write probe `.mjs` files into `/tmp/` and run them from there** — Node module resolution walks up the cwd, and `ws` will resolve to a non-existent `/private/tmp/node_modules/ws`. Always run from inside `great-banach/`.
